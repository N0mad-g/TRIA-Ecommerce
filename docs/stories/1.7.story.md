## Story

**As a** sistema,
**I want** purgar automaticamente leads com mais de 12 meses de criação,
**so that** cumpra a política de retenção definida (LGPD).

## Acceptance Criteria

1. Rota `/api/cron/purge-leads` apaga registros de `leads` com `createdAt` há mais de 12 meses.
2. Rota protegida por secret de cron (header validado) — não é endpoint público, mesmo sem sessão de usuário.
3. Job agendado via Vercel Cron Jobs (nativo, mensal) — sem infraestrutura adicional (NFR7).
4. Execução registrada em log com contagem de registros apagados, para auditoria.

### API Specification

```yaml
GET /api/cron/purge-leads
security: [cronSecret]  # header Authorization
```

### Function Template (código de referência)

[Source: architecture.md Seção 9.1 — já mostrado integralmente]

```typescript
// app/api/cron/purge-leads/route.ts
export async function GET(req: NextRequest) {
  const authHeader = req.headers.get('authorization');
  if (authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
    return NextResponse.json({ error: { code: 'UNAUTHORIZED' } }, { status: 401 });
  }
  const deletedCount = await purgeInactiveLeads();
  console.log(`[purge-leads] ${deletedCount} registros apagados`);
  return NextResponse.json({ deletedCount });
}
```

### Technical Constraints

- **Verificação de secret é o primeiro passo, sempre** — mesma regra crítica do webhook Stripe (Coding Standards regra 1), aplicada aqui a um secret diferente (`CRON_SECRET`, não assinatura Stripe), mas o princípio é idêntico: nunca processar antes de autenticar.
- **`DELETE` físico, não soft-delete** — a decisão de negócio (PRD Seção 2.5) é purga real, não `deleted_at`. Não introduzir soft-delete aqui sem confirmar com o produto.
- **Service role, não anon key** — só o Cron (autenticado via secret) pode apagar; a policy RLS de `leads` não dá `DELETE` a `anon`/`authenticated` [Source: schema-design.md Seção 8].

### Testing

[Source: architecture.md Seção 14]

- `purgeInactiveLeads()`: teste de integração confirmando que leads com `created_at` > 12 meses são apagados, e leads mais recentes **não** são.
- Rota: teste confirmando `401` sem secret válido, e que nenhum `DELETE` é executado nesse caso (mock não invocado) — mesmo padrão do teste de assinatura de webhook (Architecture Seção 14.3).
