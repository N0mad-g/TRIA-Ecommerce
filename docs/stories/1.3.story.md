## Story

**As a** visitante,
**I want** ver a Home com hero carrossel, seção de marca, benefícios, vitrines de protocolo/produto e depoimentos,
**so that** eu entenda a marca e possa navegar até Assinatura ou Produtos.

## Acceptance Criteria

1. Hero renderiza 3 slides com autoplay, navegação manual e dots.
2. Seção "Por que TRIA", 3 pilares de benefício e depoimentos renderizam com conteúdo do brief.
3. Vitrine de Protocolos exibe os 3 protocolos com "Ritual de Autoridade" destacado, linkando para `/assinatura`.
4. Vitrine de Produtos avulsos exibe amostra linkando para `/produtos`.
5. Layout responsivo mobile-first, validado em viewport mobile e desktop.
6. Formulário de captura de lead (slide 3 do hero) valida e-mail ou WhatsApp no client, envia para `POST /api/leads` (endpoint público) e persiste no banco. Exibe confirmação de sucesso e mensagem de erro em caso de falha de validação/rede.
7. Formulário exibe checkbox de consentimento LGPD (texto: aceite de uso do contato para comunicação da TRIA), **não pré-marcado** — botão de envio permanece desabilitado até o checkbox ser marcado.

### API Specification — `POST /api/leads`

[Source: architecture.md Seção 5 — API Specification]

```yaml
POST /api/leads
security: []  # endpoint público — mitigado por rate limit no Vercel Firewall
body: { contact: string, contactType: 'email'|'whatsapp', utmSource?: string, utmCampaign?: string, consentGiven: boolean }
responses:
  201: Lead persistido
  400: contact/contactType/consentGiven inválido
  429: Rate limit excedido (Vercel Firewall — infra, não código desta story)
```

### Data Model — Lead

[Source: architecture.md Seção 4.5]

```typescript
interface Lead {
  id: string;
  contact: string;
  contactType: 'email' | 'whatsapp';
  utmSource: string | null;
  utmCampaign: string | null;
  consentGiven: boolean;
  createdAt: string;
}
```

### LGPD — Consentimento (crítico, AC 7)

[Source: docs/prd.md Seção 2.5]

- Checkbox **nunca** pré-marcado.
- Botão de envio desabilitado até o checkbox ser marcado — validação client-side E `CHECK (consent_given = true)` no banco (defesa em profundidade, Story 1.2/schema-design.md Seção 4).
- Texto do checkbox: aceite de uso do contato para comunicação da TRIA (copy exata é responsabilidade do time de conteúdo, não bloqueia esta story).

### Accessibility (WCAG AA, requisito confirmado — não opcional)

[Source: architecture.md Seção 8.5]

- Hero Carousel: `aria-roledescription="carousel"`, dots com `aria-label`/`aria-current`.
- Formulário: estado de sucesso/erro em região `aria-live="polite"`.
- Todo elemento clicável é `<button>`/`<a>` semântico — nunca `<div onClick>`.
- Teste `jest-axe` obrigatório nos componentes novos desta story (Coding Standards regra 9).

### Technical Constraints

- **ISR, não SSR:** `revalidate: 3600` — não usar `force-dynamic` nesta página [Source: architecture.md Seção 2.5].
- **Rota `/api/leads` usa anon key + RLS**, não service role — diferente do padrão do webhook (Story 2.2), decisão deliberada de defesa em profundidade [Source: architecture.md Seção 8].

### Testing

[Source: architecture.md Seção 14]

- Componente: `HeroCarousel.test.tsx`, `LeadForm.test.tsx` (Jest + RTL) — incluir asserção `jest-axe` em ambos.
- API: `app/api/leads/route.test.ts` — valida rejeição de `consentGiven: false` (400, sem tocar no banco), valida sucesso (201) com dado válido.
