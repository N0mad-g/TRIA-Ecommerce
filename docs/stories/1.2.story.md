## Story

**As a** dev,
**I want** o schema Supabase para protocolos e produtos (preços, ativos, imagens, métricas de prova social) criado e populado com os dados reais do brief,
**so that** as páginas renderizem dados reais em vez de mocks.

## Acceptance Criteria

1. Tabelas `products`, `protocols` e `protocol_products` criadas no Supabase com campos para nome, preço avulso, preço assinatura, categoria, ativos, imagem, métricas de prova social.
2. Seed contém os 3 protocolos e 5 produtos avulsos do brief com valores corretos, incluindo os relacionamentos N:N corretos em `protocol_products`.
3. Camada de dados (`lib/data/catalog.ts`, Supabase client) lê e retorna os dados seedados corretamente.
4. Nenhuma referência à palavra "Kit" nos dados seedados.

### Data Models (relevante a esta story)

[Source: architecture.md Seção 4.1/4.2 — Product/Protocol]

```typescript
interface Product {
  id: string;
  slug: string;
  name: string;
  category: 'cabelo' | 'barba';
  volume: string;
  priceCents: number;
  activeIngredients: string[];
  stripePriceId: string;
  imageUrl: string;
  socialProof: { rating: number; customerCount: string; resultPercentage: number };
  relatedProtocolIds: string[];
}

interface Protocol {
  id: string;
  slug: string;
  name: string;
  productIds: string[];
  monthlyPriceCents: number;
  annualPriceCents: number;
  oneTimeEquivalentPriceCents: number;
  isFeatured: boolean;
  stripePriceIdMonthly: string;
  stripePriceIdAnnual: string;
}
```

**Regra de negócio importante:** `relatedProtocolIds`/`productIds` NÃO são colunas — são resolvidos via `protocol_products`, consultada nas duas direções (`WHERE protocol_id = ?` e `WHERE product_id = ?`). Não criar duas tabelas de junção — já foi um erro corrigido nesta sessão (Architecture 7.1). [Source: architecture.md Seção 7.1]

### Technical Constraints

- **Escopo desta migration é só catálogo** — `orders`/`subscriptions` (Story 2.2) e `leads` (Story 1.3) ganham migrations próprias depois, no commit da story que primeiro escreve nelas. Não antecipar essas tabelas aqui [Source: schema-design.md Seção 10/14 — corrigido nesta sessão, era autocontraditório antes].
- **`stripe_price_id*` sem valor real ainda** — Story 2.1 é quem cria os Products/Prices no Stripe e preenche esses campos. Usar `NULL`/placeholder por enquanto (colunas devem permitir isso temporariamente, ou a migration desta story usa constraint `NOT NULL` — se assim for, o seed desta story precisa de um valor placeholder textual óbvio, ex: `'pending_stripe_setup'`, nunca string vazia).
- **Fonte dos dados do seed:** catálogo real (5 produtos, 3 protocolos, preços, ativos, composição de cada protocolo) está no Project Brief original fornecido pelo usuário no início da sessão de PRD — não reinventar valores.

### Testing

[Source: architecture.md Seção 14]

- Teste de integração: `lib/data/catalog.test.ts` — `getProducts()`/`getProtocols()` retornam os valores seedados corretos (contagem, preços, categoria).
- Teste de constraint: seed que violasse um `CHECK`/`UNIQUE` deve falhar a migration — validar manualmente que os dados reais passam antes de aplicar em qualquer ambiente.
